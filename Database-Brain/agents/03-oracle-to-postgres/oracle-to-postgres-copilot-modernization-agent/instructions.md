# oracle-to-postgres-copilot-modernization-agent

## Role

Drive a **full-stack Oracle → PostgreSQL modernization** (database + application code) using the
**official Microsoft 2026 toolchain**:

1. **PostgreSQL extension for Visual Studio Code** (`ms-ossdata.vscode-pgsql`) — connection,
   schema browse, query editor, **Oracle → PG schema migration**, and generation of `coding_notes.md`.
2. **GitHub Copilot App Modernization for Java** — assesses a Java codebase, identifies Oracle usage,
   runs the **"Migrate from Oracle DB to PostgreSQL"** task that refactors SQL dialect + switches
   JDBC authentication to **Managed Identity**, using `coding_notes.md` as semantic guidance.

This agent is the **GUI / Copilot-driven path**. The CLI / scriptable path lives in
[oracle-to-postgres-migration-agent](../oracle-to-postgres-migration-agent/instructions.md) (Ora2Pg).

## When to use this agent vs the Ora2Pg agent

| Use this Copilot agent | Use the Ora2Pg agent |
|---|---|
| App is **Java** (Spring Boot, Quarkus, Jakarta EE) | App is .NET, Python, Node, COBOL, or no app at all |
| Demo / POC for a prospect — visual, narrative | Production CI/CD — scriptable, reproducible |
| You want **`coding_notes.md`** to guide Copilot's SQL conversion in app code | DB-only migration, no application refactor |
| Audience: cloud architects, developers, decision-makers | Audience: DBAs, data engineers, SRE |
| Showcase MS first-party tooling story | Showcase open-source / portable approach |
| Heterogeneous schema sizes < ~500 objects | Very large schemas with heavy PL/SQL, complex packages |

**Recommended combo for grand-compte demos**: run Ora2Pg first for the bulk schema + data move
(faster, parallelizable), then use this agent for the Java application refactor — Copilot still
benefits from `coding_notes.md` generated by the VS Code extension.

## Scope

| In scope | Out of scope |
|---|---|
| PostgreSQL VS Code extension setup + Entra ID connection | Linux GUI installation (use Windows / macOS / WSL) |
| Oracle schema → PG schema conversion via extension wizard | Heavy PL/SQL packages (>500 LOC) — manual review |
| `coding_notes.md` generation, review, hand-edit | Custom AI fine-tuning of Copilot |
| Java app assessment + "Migrate from Oracle to PG" task | Non-Java apps (use Copilot generic refactor) |
| JDBC → Managed Identity switch in `application.properties` / `application.yml` | Spring Cloud Config, secret rotation policy |
| Demo storyline: Oracle XE source → PG Flex target → Spring PetClinic Oracle fork | App deployment to App Service / Container Apps (separate agent) |

## Hard rules

1. **VS Code version >= 1.95** required for the PG extension preview features. Check `code --version`.
2. **Extension identifier is `ms-ossdata.vscode-pgsql`** — there are several third-party "PostgreSQL"
   extensions; only the Microsoft one supports the Oracle → PG migration wizard + Copilot integration.
3. **GitHub Copilot subscription** (Business or Enterprise) required — Free tier does NOT expose
   the App Modernization task surface.
4. **App Modernization extension**: install `vscjava.vscode-java-pack` + `ms-azuretools.vscode-bicep`
   + the **App Modernization for Java** extension (look for "GitHub Copilot app modernization" in Marketplace).
5. **Always commit `coding_notes.md`** to source control under
   `.github/postgre-migrations/<scenario>/results/application_guidance/coding_notes.md` — Copilot reads
   it from this exact path. Custom locations are NOT picked up.
6. **Entra ID, not password** for the Java app connection — `application.properties` MUST use
   `azure.postgresql.passwordless.enabled=true` after migration. If Copilot leaves a hardcoded password,
   reject the diff and ask Copilot again with explicit "use Managed Identity" prompt.
7. **PG Flex must have `Microsoft Entra Authentication` enabled** AND the app's Managed Identity must
   be added as PG admin BEFORE the Copilot task runs — otherwise the generated code compiles but
   throws `PSQLException: FATAL: no pg_hba.conf entry` at runtime.
8. **Sample Java app for demos**: use `spring-petclinic` forked with Oracle JDBC dialect (community
   fork `spring-petclinic-oracle`). Avoid building your own — Copilot is well-trained on PetClinic.
9. **Never run on production code without `coding_notes.md`**: Copilot falls back to generic
   Oracle→PG syntax rules and may miss schema-specific mappings (custom types, sequences).
10. **Two-pass workflow**: pass 1 = extension converts schema → notes file. Pass 2 = Copilot task
    refactors app code using notes. Skipping pass 1 degrades pass 2 quality by ~30-50%.

## Workflow (demo runbook)

### Pass 0 — Setup (one-shot, ~10 min)

1. Install VS Code extensions (see `prerequisites.md`)
2. Sign in to Azure (`Azure: Sign In` command palette)
3. Sign in to GitHub (Copilot status bar)
4. Open Java app folder (e.g. `spring-petclinic-oracle`)
5. Copy `config/recommended-vscode-settings.json` to `.vscode/settings.json`

### Pass 1 — DB migration via PG VS Code extension (~15 min)

1. Click the **elephant icon** in the activity bar → Connections panel
2. **Add Oracle connection** (extension supports Oracle as a source for the migration wizard)
   - Easy Connect: `<oracle-fqdn>:1521/XEPDB1`
   - Auth: SQL Server-style password (paste SYS or schema user)
3. **Add PG target connection**
   - Browse Azure → pick the Flex Server → Entra ID auth (no password)
4. Right-click Oracle DB → **Migrate to PostgreSQL** (wizard)
5. Select schema(s) → review type mappings → click **Convert**
6. **Output**: `.github/postgre-migrations/<scenario>/results/application_guidance/coding_notes.md`
   + a `.sql` file with PG-compatible DDL
7. **Review `coding_notes.md`** — this is the AI's view of what changed. Hand-edit if missing context.
8. Apply DDL to PG target (drag-drop into Query Editor → run)

### Pass 2 — App code refactor via Copilot App Modernization (~20 min)

1. Open Command Palette → **Copilot Modernization: Assess Java Project**
2. Wait for assessment (~2-5 min) → review report
3. Locate the **"Database Migration (Oracle)"** finding → default solution is
   *"Migrate from Oracle DB to PostgreSQL"*
4. Verify `coding_notes.md` is at the expected path (Hard rule #5)
5. Click **Run Task** → Copilot refactors:
   - JDBC URL in `application.properties`
   - SQL queries in `@Query` annotations + `*.xml` MyBatis / `*.sql` resources
   - `pom.xml` dependencies (`ojdbc` → `postgresql` + `azure-identity-extensions`)
   - Switches to `DefaultAzureCredential` + `azure.postgresql.passwordless.enabled=true`
6. **Review every diff** — accept, edit, or ask Copilot to redo with stricter prompt
7. Run app tests locally — ideally Testcontainers with `postgres:16` to validate without Azure

### Pass 3 — Deploy & smoke (~10 min)

1. Build: `./mvnw clean package`
2. Deploy to Container Apps / App Service (separate agent, out of scope here)
3. Verify Managed Identity → PG connection works (no password in env vars)
4. Hit a `/owners` endpoint that triggers a SQL query previously written for Oracle dialect
5. Compare row counts with Oracle source — should match the Ora2Pg validation results

## Common pitfalls

- **Extension can't see Oracle**: the PG extension uses native drivers under the hood. On Windows,
  install Oracle Instant Client (Basic Light) and put on PATH, otherwise the Oracle connection
  silently fails to enumerate schemas.
- **`coding_notes.md` not found by Copilot**: path is case-sensitive AND must include the
  `.github/postgre-migrations/<scenario>/results/application_guidance/` segment exactly.
  Renaming the folder breaks the link.
- **Copilot rewrites password-based connection**: if you don't enforce Hard rule #6 via prompt,
  Copilot may keep `spring.datasource.password=${PG_PWD}` — that's a regression vs the MS-recommended
  passwordless pattern. Always re-prompt: *"Use Azure Managed Identity, no password."*
- **PG Flex rejects Entra at runtime**: enable Entra auth on the Flex Server AND add the app's
  Managed Identity as `Microsoft Entra admin` (or grant via `pg_aad_user`). The Copilot task
  doesn't do this for you — it's an Azure config step done in portal / Bicep.
- **`@Query(value = "SELECT ... FROM DUAL")`**: PG has no `DUAL`. Copilot usually rewrites to bare
  `SELECT ...` but in JPA-native queries with vendor lock-in detection it may leave it. Search
  codebase for `DUAL` post-migration.
- **Oracle `ROWNUM` / `ROWID`**: Copilot converts `ROWNUM <= N` to `LIMIT N` correctly but
  `ORDER BY ... FETCH FIRST N ROWS ONLY` translation needs review for tie-breaking semantics.
- **JSON columns**: Oracle `JSON` ↔ PG `JSONB` — Copilot picks `JSONB` (correct), but Spring's
  JPA mapping needs `@JdbcTypeCode(SqlTypes.JSON)` instead of `@Type(type = "json")`. The task
  may not catch all annotations.
- **Sequences in `@GeneratedValue`**: Oracle `SEQUENCE.NEXTVAL` becomes PG `nextval('seq_name')`.
  Sequence ownership and increment-by-1 vs cache=20 differences may surface as PK conflicts after
  `setval()` from Ora2Pg's data load. Always run the post-load fixup from
  [`../oracle-to-postgres-migration-agent/scripts/post-load.sql`](../oracle-to-postgres-migration-agent/scripts/post-load.sql).

## Lessons from the docs (MS Learn, 2026-03)

- "Coding notes detail schema changes... that you can **optionally** use to produce more precise
  and semantically aligned PostgreSQL-compatible code" — translation: skip them and you get
  generic Oracle→PG. Always generate.
- The migration task explicitly addresses **two concerns simultaneously**: SQL dialect AND
  Managed Identity authentication. Don't ask Copilot to do them separately — the task handles both.
- Coding notes can include: data type mappings, sequences/identities/composite types, date/time
  adjustments, referential integrity references, **PL/SQL package summaries with procedure / function
  signatures**, and AI-generated translation hints.

## Validation checklist

- [ ] `coding_notes.md` present at the expected path and reviewed
- [ ] PG target schema applied — `\dt` in psql shows all expected tables
- [ ] Java app builds (`mvn package`) with no Oracle JDBC dependency in `pom.xml`
- [ ] Local test pass (Testcontainers `postgres:16`)
- [ ] No `password=` in `application.properties` for the PG datasource
- [ ] Managed Identity confirmed via `az postgres flexible-server execute --query "SELECT current_user;"`
- [ ] Functional smoke: same endpoint returns same data on Oracle and PG sides
