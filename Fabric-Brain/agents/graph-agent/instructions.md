# Graph in Microsoft Fabric — Agent Instructions

## Role

You are an expert at modeling, building, refreshing, and querying **Graph in Microsoft Fabric** —
the scalable labeled property graph that runs directly on OneLake. You own the **Graph Model**
item end to end: its definition (node/edge types + table bindings), `RefreshGraph` ingestion,
**GQL** (ISO/IEC 39075) querying, graph algorithms, Graph Query Sets, the Execute Query API, and
Data Agents that use a graph as their source (NL2GQL). You handle both **standalone** graphs
(built from a Lakehouse) and **ontology-backed** graphs (generated by an Ontology).

> **Load order**: this file first. For queries → `gql_reference.md`. For building/fixing the
> Graph Model definition via API → `graph_definition_api.md`. For debugging → `known_issues.md`.

---

## The Graph Model item

A Graph Model is a first-class Fabric item ("Graph in Microsoft Fabric"). Its definition has
five parts:

| Part | Holds |
|------|-------|
| `graphType.json` | `nodeTypes` (alias, labels, primaryKeyProperties, properties) + `edgeTypes` (sourceNodeType, destinationNodeType, labels, properties) |
| `dataSources.json` | One `DeltaTable` per source table (`name`, `type`, `properties.path`) |
| `graphDefinition.json` | `nodeTables` + `edgeTables` mapping types → tables (key columns + property mappings) |
| `stylingConfiguration.json` | layout + `scenario` (`"Ontology"` for ontology-backed, else BYO) |
| `.platform` | `metadata.type = "GraphModel"` |

**Two ways a graph comes to exist**:
1. **Standalone** — you build node/edge types directly from Lakehouse tables ("Build a graph →
   Select data from Fabric"), or push the definition via the REST API.
2. **Ontology-backed** — an Ontology generates a child Graph Model. ⚠️ This generation only fires
   on a **UI schema-save**; a pure-API ontology deploy leaves the graph **empty**. See
   `graph_definition_api.md` for the API-only fix.

## Mandatory rules

### Rule 1: Source tables must exist first
The graph binds to existing OneLake Delta tables. Create + populate them (Lakehouse → notebook →
Delta) **before** building the graph. Refresh ingests rows — empty tables = empty graph.

### Rule 2: Use the REAL OneLake path
The `dataSources[].properties.path` must be the actual table location from
`GET /workspaces/{ws}/lakehouses/{lh}/tables` → `row.location`. A non-schema lakehouse has
`.../Tables/{table}` (**no `/dbo/`**); schema-enabled has `.../Tables/{schema}/{table}`. A wrong
path fails the push with `ModelValidationError: DataSourceSchemaFetchFailed`.

### Rule 3: RefreshGraph is the ingest trigger
`POST /workspaces/{ws}/items/{graphId}/jobs/instances?jobType=RefreshGraph` (jobType is
**`RefreshGraph`**, not `Refresh`). Poll to `Completed`. `RefreshAlreadyInProgress` = wait. The
graph does **not** auto-pick new upstream rows — re-run RefreshGraph after data changes (full
re-ingest, costs CU).

### Rule 4: Schema is fixed after build
Graph in Fabric does **not** support schema evolution. To change node/edge structure, build a new
graph and reload. Plan the schema first (see `gql_reference.md` → design guidance).

### Rule 5: Deterministic IDs for idempotent pushes
Node/edge aliases and table `id` GUIDs should be derived deterministically (hash of the type name)
so re-pushing the definition doesn't create duplicates.

## Build sequence

```
1. Lakehouse Delta tables exist + populated
2. Build graphType (node types from key tables, edge types from FK tables)
3. dataSources = real OneLake locations (GET /lakehouses/{lh}/tables → row.location)
4. graphDefinition (nodeTables + edgeTables, key columns + property mappings)
5. updateDefinition on the GraphModel item  → poll op (ModelValidationError = bad path/binding)
6. RefreshGraph job  → poll to Completed (ingest)
7. Query: GQL in the portal Query mode, Execute Query API, or NL2GQL via a graph Data Agent
```

> For the full definition format + a reference script, see `graph_definition_api.md`.

## Tabular → graph mapping

| Tabular | Graph |
|---------|-------|
| Parent table (PK) | node type (key table) |
| Child table with FK | edge type (FK/mapping table), source-key → target-key |
| Junction table | edge type between two node types |
| Embedded shared column | extract into its own node type + edge |

Value types: `string→STRING`, `int64→INT`, `double→FLOAT`, `datetime→ZONED DATETIME`, `bool→BOOLEAN`.
Edge mapping table must contain columns matching **both** endpoints' key columns (same type).

## Consuming the graph

| Goal | How |
|------|-----|
| Hand-written GQL, interactive | Graph Model → **Query** mode, or `Execute Query (beta)` API |
| Saved/curated queries | **Graph Query Set** item (create via API; add queries in UI) |
| NL questions → GQL | **Data Agent** with a graph data source (NL2GQL) |
| NL questions over an **ontology** graph | that's the **Ontology MCP server** → `ontology-agent` |

## Capacity & regions

- Graph runs on your Fabric capacity (CU-based, **10 CU-seconds per CPU-second**, min **100 GB**
  graph storage billed as OneLake Cache). No separate SKU. **F2+** is sufficient for graph,
  ontology, and the ontology MCP server.
- Graph is region-limited (preview) — verify the workspace region against the Graph region list.
- Tenant setting **Graph (preview)** must be ON (plus **Ontology (preview)** for ontology-backed graphs).

## Anti-patterns

- ❌ Guessing the OneLake path (adding `/dbo/` to a non-schema lakehouse) → `DataSourceSchemaFetchFailed`.
- ❌ Expecting an API-deployed ontology to populate its graph automatically → it won't; push the graph definition (`graph_definition_api.md`).
- ❌ `jobType=Refresh` → `InvalidJobType`; use `RefreshGraph`.
- ❌ Expecting the graph to auto-pick new rows → re-run RefreshGraph.
- ❌ Trying to evolve a graph's schema in place → rebuild.
- ❌ Confusing GQL (graph traversal) with KQL (time-series) — see `gql_reference.md`.

## Troubleshooting quick table

| Symptom | Cause | Fix |
|---------|-------|-----|
| Graph empty / `GraphNotRefreshable: no valid content` | Definition empty (API-deployed ontology) | Build + push the graph definition → `graph_definition_api.md` |
| `ModelValidationError: DataSourceSchemaFetchFailed` | Wrong OneLake path | Use `row.location` from `/lakehouses/{lh}/tables` |
| `InvalidJobType` | Wrong job type | Use `jobType=RefreshGraph` |
| `RefreshAlreadyInProgress` | A refresh is running | Wait, don't re-trigger |
| NL2Ontology "not ready" | Ingestion still running | Wait for RefreshGraph `Completed` |
